## A custom project should begin with a stable problem.

Automation does not repair a process nobody understands. A dashboard does not create a useful decision merely by collecting more numbers. An integration does not become maintainable simply because two products can exchange data.

Before recommending custom work, Boho should be able to answer:

- What repeated task or decision is being improved?
- Who owns the process?
- Who uses the system?
- What are the inputs?
- What output is required?
- Which systems and permissions are involved?
- What happens when the system fails?
- Where must a person review or approve the result?
- Who maintains it?
- What is the exit or export path?

When those answers do not exist, the first useful project may be process clarification rather than software.

## This service is a good fit when

- A repeated business task consumes meaningful time or creates avoidable errors.
- Existing software almost solves the problem but leaves a specific operational gap.
- Data must move between approved systems in a controlled and observable way.
- A focused calculator, form, dashboard, workflow, or publishing tool would improve a clear decision or customer action.
- The inputs, outputs, owner, permissions, and maintenance responsibilities can be defined.
- The business values documentation, portability, and visible failure behavior.
- The project can be divided into a bounded first release.

## This service may not be the right fit when

- The underlying process changes every week.
- Nobody owns the workflow or approves decisions.
- The requested automation must act without appropriate human review.
- The system would require credentials, permissions, data, or actions the client is not authorized to provide.
- A standard supported product would solve the problem more economically.
- The project requires continuous support, regulated assurance, safety-critical operation, or specialist security work not included in the scope.
- The request depends on misleading users, bypassing platform rules, fabricating content, or concealing automated behavior where disclosure is required.

## The custom-project process

### 1. Define the useful outcome

Boho and the client define the repeated task, decision, customer action, or operating problem the system should improve.

### 2. Map the workflow and permissions

Users, systems, data, inputs, outputs, access, approvals, failure states, and maintenance responsibilities are documented.

### 3. Choose the smallest responsible approach

Boho compares a custom build with existing products, configuration, integration, manual process improvement, or a smaller experiment.

### 4. Approve the scope

The written proposal identifies the first release, price, timing, deliverables, exclusions, dependencies, infrastructure, ownership, licensing, support, and acceptance criteria.

### 5. Build and test

The system is implemented in bounded stages. Relevant functions, permissions, data handling, errors, retries, notifications, logs, and user actions are tested according to the approved scope.

### 6. Deploy and document

The final record explains deployment, providers, accounts, ownership, configuration, operating instructions, known limitations, maintenance, support, recovery, and next actions.

### 7. Maintain only where necessary

Ongoing hosting, monitoring, support, data operations, provider administration, or later development require a separately stated recurring scope unless included in the original engagement.

## Security, privacy, and human review

Custom systems can create risk when they move data, modify accounts, publish content, send messages, or automate decisions.

The written scope should identify:

- What data is collected or processed
- Why it is needed
- Where it is stored
- Who can access it
- How long it is retained
- Which providers receive it
- Which actions require human approval
- What logs are kept
- What happens when a provider fails
- How credentials are stored
- How access is removed
- How the system is backed up or exported
- Which risks require specialist review

Boho should not autonomously send outreach, spend money, publish claims, delete assets, change production accounts, expose private information, or perform another consequential external action without the approved permissions and human-review boundaries stated in the engagement.

## Ownership, licensing, and portability

The proposal should state:

- Which code and deliverables the client owns
- Which pre-existing Boho materials remain licensed
- Which open-source or third-party licenses apply
- Which accounts the client owns
- Which providers and recurring fees are involved
- Whether source code is delivered
- Whether deployment scripts and documentation are included
- What happens when support ends
- How data and configuration can be exported
- Which parts of the system depend on a third-party platform

Boho generally prefers client ownership of durable business assets and clear exit treatment. Platform, license, and provider limits must still be disclosed.

## Maintenance and support

A custom build is not automatically a lifetime maintenance agreement.

The written engagement should state:

- Defect-remediation or warranty period, if any
- What counts as a defect
- What counts as a new requirement
- Supported providers and versions
- Monitoring responsibilities
- Backup and recovery responsibilities
- Security-update responsibilities
- Support channels and hours
- Response commitments, if any
- Usage or volume limits
- Ongoing provider costs
- Change-request process
- End-of-support and offboarding terms

No custom system, provider, API, hosting platform, or network is outage-proof. Any binding service level must be expressly stated in the written engagement.

## Frequently asked questions

### How do I know whether we need a custom build?

Start with the business process. If a stable repeated task has a clear owner, known inputs, defined outputs, meaningful consequences, and a gap that standard products cannot economically solve, custom work may be justified. Discovery can answer that question before a larger build is approved.

### What can be built for the $2,500 minimum?

A small, focused tool or simple integration with one primary workflow, limited systems or users, bounded inputs and outputs, standard deployment, testing, and documentation. Larger interfaces, more integrations, sensitive data, complex permissions, high volume, or ongoing support require a different quote.

### Is discovery required?

Not always as a separate engagement. A very clear small project may be quotable after the free review and discussion. Discovery is required when the workflow, systems, permissions, data, risks, or correct solution are not yet clear enough for a responsible build quote.

### Is the discovery fee credited toward the build?

Yes, when it is eligible, the approved build concerns substantially the same system, the proposal is accepted within 90 days, and the published conditions are met.

### Will we own the code?

The proposal states ownership and licensing. Boho generally prefers clear client ownership of project-specific durable assets while retaining ownership or licensing terms for pre-existing tools, reusable methods, open-source components, and third-party services.

### Can you integrate with our existing software?

Potentially, after reviewing the provider’s API, authentication, permissions, rate limits, data terms, reliability, costs, and support. A vendor’s marketing claim that an API exists does not guarantee that it supports the required workflow.

### Do you build AI automation?

Boho may build approved AI-assisted workflows where the task, data, review boundary, failure risk, provider terms, cost, and human responsibility are clear. Boho does not recommend unreviewed automation for consequential publishing, account changes, sensitive decisions, or claims.

### Can you handle sensitive or regulated data?

Only after a specific review of the data, law, contracts, security requirements, providers, permissions, retention, and specialist obligations. Ordinary public pricing does not assume regulated or high-risk data.

### Does the build include hosting?

Only when the written scope says so. Custom systems often require separately priced infrastructure, provider usage, monitoring, backups, and maintenance.

### What happens when an API or provider changes?

The system may require maintenance or modification. The proposal should state supported dependencies, monitoring, update responsibilities, and whether ongoing support is included.

### Can you automate sending, publishing, deleting, or account changes?

Only with explicit authorization, appropriate permissions, logging, error handling, and human-review boundaries. High-consequence actions should not be treated as harmless background automation.

### What happens after launch?

Boho provides the agreed documentation and handoff. Maintenance, monitoring, provider administration, support, feature development, and later integrations continue only under a separately approved scope unless already included.
